Nodebeats Documentation

Nodebeats is an Open Source Content Management System built using MEAN Framework that allows you to create web-applications with ease. It is built on top of a RESTful JSON api. You can easily create custom modules on top of this Content Management System and quickly launch your web application.

After you have a copy of this software on your local disk, As always, the first step is to install all the dependencies that this application requires.

From the root of your project directory, run the command:
npm install

This will install all the dependencies that are required by the project. Before running it, you have to create database for the application and some users for database authentication purpose.

You have two choices in terms of MongoDB database. Either, You can use local instance of MongoDB database or choose one of the Database-as-a-Service providers for MongoDB. mlab is one of the most popular Database-as-a-Service provider for MongoDB.

So, let's get started with MongoDB database for the application.

Local Instance of MongoDB

Obviously,the first thing you might want to do is create and configure the MongoDB database on your local machine. That is exactly what you are going to do now.

Follow the following step-by-step guidelines to create and configure MongoDB in local machine.

  1. The first thing, you would want to do is enable access control in MongoDB database because of security risks. Before enabling role based access control, you should first consider the users of the system. So, create a user with userAdminAnyDatabase role access in admin database, which you will use later to login and create users for any database and then enable authentication.

    To create a user for admin database, open the terminal and run mongo command.
    mongo

    Run the following command to create user administrator for your mongodb database instance with userAdminAnyDatabase role access with your own unique and strong password:
    use admin 

    											
		db.createUser({
			"user":"superadmin",
			"pwd":"superadmin#123",
			"roles":[{
				"role":"userAdminAnyDatabase",
				"db":"admin"
			}]
		})

    This will successfully create user administrator for the mongodb database instance.

  2. After creating user administrator, enable the access control feature in mongodb database server. To enable it, run the following command with sudo access in ubuntu.


    sudo nano /etc/mongod.conf


    You will be prompted for administrative password. Enter your password and press enter. Navigate to security section and uncomment both lines. 


    								
			security:
			authorization: enabled

    And finally save it before you exit the terminal.

  3. Restart the mongodb server instance


    sudo service mongod restart

  4. Now run the mongodb instance with newly created user


    mongo --host localhost --port 27017 -u "superadmin" -p "superadmin#123" --authenticationDatabase "admin"

  5. Now create a database. To create database, enter following command


    use dbName


    This will create database with specified name. Now it's time to add users to it.

  6. Create two users for the database. One will be database admin user and other will be user with readWrite access only.

    You will use this user for accessing the database from terminal. So, Create user with dbAdmin role: 

    									
			use dbName


			db.createUser({
					"user":"nodebeatssuperadmin",
					"pwd":"nodebeatssuperadmin#123",
					"roles":[{
					"role":"dbAdmin",
					"db":"prj_nodebeats"
				},{
					"role":"userAdmin",
					"db":"prj_nodebeats"
				},{
					"role":"dbOwner",
					"db":"prj_nodebeats"
				}]
			})

    Following is the user which you will use to connect to the database from web application. 

    					
				use dbName


				db.createUser({
					"user":"nodebeatsadmin",
					"pwd":"nodebeatsadmin#123",
					"roles":[{
						"role":"readWrite",
						"db":"prj_nodebeats"
					}]
				})

So, You now have a database with some users on it. Copy the username, password and database name. you will need that later when running the application.

Deploying to the cloud:

Before, deploying project to the cloud, you have to do certain things like creating backup of the database so that you can restore the database archive file directly to the prouction MongoDB server. Likewise, it's good idea to generate shrinkwrap.json file before deploying.


Creating a backup archive file of MongoDB database:
Before deploying the project to the cloud, create a backup of your mongodb database in archive extension format. This backup file in archive format will restore only on MongoBD version greater than v3.2

  1. To create backup of MongoDB database in archive extension format, run the following command: 

    									
	mongodump --host hostname --port porttNumber --username userName --password 	passwordEntry --db dbName --gzip –archive=filePath

    Give the appropriate file path.
    ex: 

    									
	mongodump --host localhost --port 27017 --username nodebitsadmin --password 	nodebitsadmin#123 --db prj_nodebeats --gzip --archive="/home/lakhe/Music/production test 	files/prj_nodebeats.archive"

  2. Now you have archive file, Open the another terminal and run the following command to restore the database file in the production MongoDB server. 

    									
	mongorestore --host hostname --port portNumber --username userName --password passwordEntry  --gzip --archive=filePath --db dbName

Note:
That's it, Now when you are ready to deploy the project in the cloud you can restore the database directly in the production server.

Creating npm-shrinkwrap.json file:
Next, if you have changed or updated the npm packages, you need to recreate the npm-shrinkwrap.json file. The npm shrinkwrap command lets you lock down the versions of a package's dependencies so that you can control exactly which versions of each dependency will be used when your package is installed. This is important because when installing the packages using npm install command, there is no guarantee that same exact package version will be installed and this may bring bugs to the application. In production environment, It may create unwanted problems and risks.

So, recreate the npm-shrinkwrap.json file if you have updated or installed the new packages.

Deploying the project to OpenShift:

So, we are deploying our application to the RedHat Openshift cloud. Openshift stands out among all the other free Node.js hosts by offering a no-trial, truly free Node.js hosting platform. So, let’s get started.

  1. Go to https://www.openshift.com/ and sign-up for new account.

  2. Complete the registration process by clicking on the confirmation link sent to your email.

  3. Login with correct credentials. After that, you will be redirected to the Openshift welcome page. That’s it for now in Openshift web interface. We will be doing everything from the terminal using rhc client tools provided by Openshift.


Openshift provides you with web interface as well as command line tools to deploy your application. For this deployment, we won't be using web interface. Rather, we will create and deploy our application project on openshift using client tools i.e. rhc. So, first of all you need to install rhc client tools.

Before deploying your code to the openshift, you still need to make some changes to the code, specifically server.js file. So, edit server.js file with the below guidelines.

  • Add following line of code at the variable declaration section along side port number variable declaration. Remove existing port variable declaration and paste the below content. 

    									
ipAddress = process.env.OPENSHIFT_NODEJS_IP,
port = process.env.OPENSHIFT_NODEJS_PORT || 8080

  • Then add these lines of code inside of setupVariables method 

    								

self.ipaddress = ipAddress;
self.port      = port;

if (typeof self.ipaddress === "undefined") {
	//  allows us to run/test the app locally.
		self.ipaddress = "127.0.0.1";
};

  • Modify the server listening section of the code 

    								
server.listen(self.port, function() {
	console.log('Node server started on' + self.port + ' at ' + Date(new Date()));
        });

    to 
    								
server.listen(self.port, self.ipaddress, function() {
	console.log('%s: Node server started on %s:%d ...',
    Date(Date.now() ), self.ipaddress, self.port);
});

    That’s it, You are now ready to deploy your code to the Openshift cloud platform.

Nginx setup in Openshift

Now that you have created and deployed your web application successfully, it's time to set-up Nginx as our proxy server that will serve all the static contents.

You have to know that we cannot setup Nginx alongside our Node.js web application because Openshift doesn't allow it. So, we have to create seperate application and then proxy the client request back to our Node.js application. Nginx allows us to create upto three small basic gears with the free account. So, let's get started:

Note:
Application name must contain only alphanumeric characters (a-z, A-Z, or 0-9).

We will be using Openshift Open-Source Nginx cartridge which you can find using following link:
https://github.com/gsterjov/openshift-nginx-cartridge

This Nginx cartridge installs Nginx v1.4.4 which is ofcourse outdated but that's not a problem because once this cartridge is installed and we have our Nginx on Openshift up and running, we can update the Nginx using our own custom configurations by compiling Nginx from source and then installing it. So, Go to the github repository page and read the instructions there.

So, Follow the following guidelines to create Nginx application on Openshift:

  • To create application, run the following command:
    rhc create-app myapp http://cartreflect-claytondev.rhcloud.com/reflect?github=gsterjov/openshift-nginx-cartridge
    This will create Nginx web server application on Openshift. Here myapp is your custom application name.

  • Now that your Nginx web server is up and running, it's time to update it.
    SSH into your application server and Navigate to data directory of app-root folder.
    cd app-root/data

    We will compile and build our Nginx server application in this directory. In Openshift server, Only data directory is the persistent one. So, If you need some data to be persistent across deployments, then, use this directory.

  • So, Follow the following guidelines to upgrade Nginx versions:

    • Create a shell script. Name it anything you want. I will name it as compile_nginx.sh

    • Go to https://gist.github.com/ShrawanLakhe/77c8d125dc4f18dbff54f72cfeb71e21 and copy the contents of the gist.

    • Clone your Nginx web application repository somewhere in the disk and place the shell script and commit the file to the repository. Push the changes to the git.

      git add compile_nginx.sh
      git commit -am "shell script committed"
      git push origin master

    • SSH into your application, and move the shell script from repo directory to the data directory of app-root folder.
      cd app-root
      mv repo/compile_nginx.sh data/

    • Now check your data directory, you will find the shell script. Now run the shell script
      cd data

    • Make the shell script executable by running the following command:
      chmod 755 compile_nginx.sh

    • To compile and install latest Nginx, run the following command
      ./compile_nginx.sh

  • The compilation process will take some time. Once done, check the nginx version
    cd nginx/usr/versions/1.11.3/bin/
    ./nginx -v

    You will see the output with latest Nginx version.

  • Now stop and then again start the nginx server

    ./nginx -s stop
    ./nginx

  • To reload nginx configurations:
    ./nginx -s reload

  • Well, that's it, You have successfully installed Nginx with latest version on Openshift.

Configuring Nginx as proxy server in Heroku

We are going to setup Nginx as a proxy server which will serve all the static resources and then redirects other requests to Node.js server in Heroku. Setting Nginx with heroku sounds tough, but actually it is very easy. We just have to understand the concept of multiple buildpacks in Heroku and how that works. There may be cases when a single buildpack is not sufficient for our application needs. Even in our case, if we want to set nginx as a proxy server, then without multiple buildpacks, we just cannot do it. So, if you want to learn more about multiple buildpacks, then follow the following links:
https://devcenter.heroku.com/articles/using-multiple-buildpacks-for-an-app


So, let’s get to the main task of setting Nginx as a proxy server. We will do it from scratch. So, follow the following step-by-step guidelines:

  1. First you need to create application. So create a directory and from inside the folder, run the following command which will create heroku app. Place all of your project files except node_modules folder in the root of this directory.
    heroku create
    If you are not logged in with heroku credentials, then first login with heroku credentials and create the application.
    heroku login

  2. Initialize the git repository in the directory
    git init

  3. Set git remote origin. To see your git remote origin,


    • Login with correct credentials to access dashboard section

    • From the dashboard page, click your app.

    • Click on Deploy tab and you will find your git remote origin there.


    ex:
    heroku git:remote -a young-citadel-30643

  4. So, it’s time to work with multiple buildpacks – first one is Node.js buildpack and another one is Nginx buildpack. There are two ways in which you can implement multiple buildpacks.

    One method involves setting buildpacks using terminal command and another one involves creating .buildpacks file inside the project root directory.

    So, you need two buildpacks. To know more about those buildpacks, Click following git repository link of heroku and search for Node.js and Nginx buildpack repository.
    https://github.com/heroku

    Or, you can follow the below links directly.

    • https://github.com/heroku/heroku-buildpack-nodejs

    • https://github.com/heroku/heroku-buildpack-nginx


    So, let’s look at both.


    Setting multiple buildpacks using terminal


    • First method involves setting buildpacks using terminal commands. So, run the following command from the project root directory.


      • heroku buildpacks:set https://github.com/heroku/heroku-buildpack-nodejs.git

      • heroku buildpacks:add https://github.com/heroku/heroku-buildpack-nginx.git

    • That’s it, Now, you need to change some code to make your application run with nginx as proxy server.

    • If you want to check what buildpacks will be used with your application, then run the following command:
      heroku buildpacks

    Setting multiple buildpacks using .buildpacks file

    • The first step is to set multi-buildpack. So run the following command:
      heroku buildpacks:set https://github.com/heroku/heroku-buildpack-multi.git

    • This method involves creating a .buildpacks file in the project root directory. That file should contain buildpacks in separate lines. That’s all there is to do.


      • touch .buildpacks

      • echo 'https://github.com/heroku/heroku-buildpack-nodejs.git' >> .buildpacks

      • echo 'https://github.com/heroku/heroku-buildpack-nginx.git' >> .buildpacks

  5. If you want to configure Nginx’s worker_processes directive via the NGINX_WORKERS environment variable, then run the following command to set your NGINX_WORKERS to 8 on a PX dyno:
    heroku config:set NGINX_WORKERS=8

  6. Now you are going to change some code from project files so that it will run without errors when deployed.


    • If you have file named Procfile in the root directory, then change the content of file to following:
      web: scripts/build_nginx.sh

      If you haven’t got that file, then create one. It’s a must have file if you want to deploy your application on heroku.

      • touch Procfile

      • echo 'web: scripts/build_nginx.sh' >> Procfile

      Do remember that, the content of Procfile must be changed after the application is deployed successfully in order to run the app successfully.

    • Following are the requirements if you want to run your application using nginx as proxy server without any errors:

      • Your webserver listens to the socket at /tmp/nginx.socket .

      • You touch /tmp/app-initialized when you are ready for traffic.

      • You can start your web server with a shell command.

    • So, let’s change our code in server.js file to meet the requirements.
      You have to first require fs module. 

      											
 var fs = require('fs');

      Replace following piece of code 

      											
	self.start = function() {
        //  Start the app on the specific interface (and port).
        server.listen(self.port, function() {
            console.log('Node server started on' + self.port + ' at ' + Date(new Date()));
        });
    };

      with

      											
	self.start = function() {
        //  Start the app on the specific interface (and port).
        server.listen('/tmp/nginx.socket', function() {
        	if (process.env.DYNO) {
			 	console.log('This is on Heroku..!!');
				fs.openSync('/tmp/app-initialized', 'w');
			}
        	console.log('Node server started on' + self.port + ' at ' + Date(new Date()));
        			});
    };

      web: bin/start-nginx node server.js

        mime.types file will be automatically created. So, if you do not want to modify this file, then it’s perfectly ok. If you do change however, then replace the content of mime.types file with the content content from this gist:
